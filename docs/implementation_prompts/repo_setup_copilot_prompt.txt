1.	pipx vs pip (why pipx was suggested)
•	pipx pros: installs CLI tools isolated from any single project, makes them globally runnable without activating a venv, and avoids polluting a project environment.
•	pip (inside a venv) pros: reproducible per-project tooling, easy to pin versions via dev-requirements.txt, better for teams who must replicate the same tool versions.
•	Recommendation for this repo: use a project venv + dev-requirements.txt (pip in venv). That gives reproducibility and matches your earlier preference for a Python environment.
2.	dev-requirements.txt (suggested content) and commands

3.	Can ruff be configured to prevent running in format mode?
•	Short answer: No — ruff has no configuration option that disables its formatting subcommand. If the binary is present, someone can invoke ruff format / ruff --fix.
•	Practical controls you can enforce:
•	CI enforcement: CI only runs ruff check . and fails the build on violations.
•	Pre-commit / Git hook: add a pre-commit hook that runs ruff check and rejects commits if the working tree has unexpected modifications (this will block commits that result from running format/fix).
•	Wrapper script: provide scripts/lint.ps1 and avoid exposing ruff directly in PATH; encourage developers to use the script.
•	Policy + docs: make the prohibition explicit in DEVELOPER_GUIDE.md and enforce in CI.
