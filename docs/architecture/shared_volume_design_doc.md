# Shared Volume Design Document

## 1. System Overview

The Climatomaton architecture relies on a strictly decoupled, socketless Interprocess Communication (IPC) model. Communication between the Core Daemon and its pluggable subsystems—the Pluggable Rules Module (PRM) and Pluggable Environment Modules (PEMs)—is facilitated entirely through file I/O operations on a shared POSIX-compliant volume.

This document defines the mechanical rules of engagement, volume topology, and the system-wide synchronization protocols required to guarantee data integrity across multiple independent processes.

## 2. The Atomic Write Protocol

To ensure the Rules Engine and IPC Broker never read partial or corrupted state data, all file modifications on the shared volume must strictly adhere to the Atomic Write Protocol.

### 2.1 Write Execution Steps

1. **Temporary File Creation:** The writing process must first write its complete JSON payload to a temporary file in the target directory. This file should typically use a `.tmp` extension or a similar process-specific suffix indicating it is in-flight.
2. **System Rename:** Once the file write and close operations are completely finalized, the process must execute an atomic OS-level rename operation (e.g., `os.rename()` in Python 3) to change the temporary file path to its final intended destination path.

### 2.2 Temporary File Cleanup

In the event of a crash or unexpected termination during a write cycle, stale temporary files may be left on the shared volume. To handle this, processes are granted explicit permission to arbitrarily remove any existing temporary files they strictly own prior to initiating a new write sequence. Components must also perform a cleanup of their owned temporary files during their respective startup (`app.initialize`) and shutdown (`app.prepare_for_shutdown`) phases.

### 2.3 Timezone Constraint

All timestamping utilized for file naming conventions, logging payloads, notifications, and internal heartbeat monitoring must strictly adhere to the **UTC timezone**.

## 3. Volume Topology

The shared volume is organized into specific directories, each dedicated to a distinct IPC function.

### 3.1 `pems/` (Environment Modules)

* **Purpose:** Stores the schemas and data states provided by active PEM processes.
* **Contents:**
  * `{pem_namespace}.schema.json`: The static JSON schema defining the environment structure and serving as the heartbeat monitor for the module (updated at an interval of no greater than 30 seconds).
  * `{pem_namespace}.json`: The active JSON environment data state.

### 3.2 `prm/` (Rules Modules)

* **Purpose:** Stores the active compiled rule logic.
* **Contents:**
  * `active_rules.json`: The compiled JSON Intermediate Representation (JSON-IR) generated by the PRM. The modification timestamp of this file acts as the PRM's heartbeat.

### 3.3 `tx/` (Transactions)

* **Purpose:** Facilitates the two-way mutation acknowledgment cycle between the Core Daemon and PEMs.
* **Contents:**
  * `req_{tx_id}_{namespace}.json`: The JSON Patch (RFC 6902) request generated by the Rules Engine.
  * `ack_{tx_id}_{namespace}.json`: The success/failure confirmation generated by the target PEM. Both files are deleted by the IPC Broker upon completion.

### 3.4 `logs/`

* **Purpose:** A drop-box for routine module logging.
* **Contents:**
  * `{timestamp}_{id}.json`: Modules write warning/error payloads here. The IPC Broker routes these to the Event Bus and immediately deletes the files to prevent disk I/O bloat.

### 3.5 `notifications/`

* **Purpose:** A drop-box for high-priority administrative alerts.
* **Contents:**
  * `{timestamp}_{id}.json`: Modules write critical alerts here. The IPC Broker translates these into `sys.notification` events for Discord delivery and immediately deletes the files.

## 4. Decentralized Schemas

While this document governs the mechanical file operations and directory structures of the shared volume, it does not dictate the contents of the files themselves. The specific JSON schema definitions for the payloads (e.g., transaction formats, PRM JSON-IR structure, notification layouts) remain strictly owned by the respective component design documents generating them.
