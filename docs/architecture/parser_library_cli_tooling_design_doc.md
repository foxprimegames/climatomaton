# Parser Library & CLI Tooling Design Document

## 1. Overview

This document outlines the architecture and specifications for the shared Python 3 parsing library and its accompanying Command Line Interface (CLI) tool. The primary function of this suite is to process Nomicron climate rules written in the plain-English Clime language (`.rules` files) and translate them into the standardized JSON Intermediate Representation (JSON-IR) expected by the Climatomaton Core Engine.

## 2. Shared Parsing Library Specifications

The core of this system is a dedicated, decoupled Python parsing library responsible for lexing, parsing, and emitting the JSON-IR.

### 2.1 I/O Decoupling Requirement

To ensure maximum portability and reusability across different components, the core library must perform absolutely zero file system I/O operations.

* **Lexing, Parsing, and Emitting:** All internal functions must be designed to accept either static memory objects (such as raw strings, lists of strings, or populated Abstract Syntax Tree objects) or memory-safe iterators yielding the appropriate textual content. Specifically, the lexer must accept any string iterator, including fixed-block-length iterators and small substring iterators. Because it accepts arbitrary iterators, the lexer is not guaranteed to receive a complete keyword or token upon receipt of a single string chunk, and must implement an internal buffer to handle elements that span chunk boundaries.
* **Outputs:** The library must return the resulting sequence of tokens, the constructed AST, or the final JSON-IR dictionary respectively, without attempting to write data to disk.

### 2.2 Error Accumulation Strategy

To provide a superior developer experience, the parser must not fast-fail or halt on the first encountered syntax error via standard exceptions.

* **Structured Results:** The parsing pipeline must implement an error recovery strategy, accumulating syntax and lexical errors as it traverses the provided input.
* **Result Classes:** The primary parsing functions must return structured objects extending an abstract base class to provide specific typing:
  * `BaseResult`: An abstract class containing a `success` field (boolean, indicating if the operation completed without fatal structural errors) and an `errors` field (a list of structured error objects containing line numbers, column numbers, and specific error messages).
  * `ASTResult`: Extends `BaseResult` and adds an `ast` field containing the constructed Abstract Syntax Tree (which may be partial or null if parsing failed entirely).
  * `IRResult`: Extends `BaseResult` and adds a `json_ir` field containing the resulting JSON-IR dictionary payload.

### 2.3 Lexical Analyzer (Lexer)

The Lexer is responsible for consuming the raw string iterator and producing a discrete stream of tokens.

* **Token Definition:** A **token** is defined as a structured data object containing the token classification type (e.g., `KEYWORD`, `IDENTIFIER`, `NUMBER_LITERAL`, `STRING_LITERAL`, `OPERATOR`, `EOF`), the extracted literal string value, and the exact line and column metadata for error reporting.
* **State Machine & Buffering:** Because the string iterator may yield incomplete substrings, and not all language tokens are whitespace-separated, the Lexer must manage a character buffer. To safely resolve fragmented tokens and distinguish between an incomplete read and a true syntax error, the Lexer will utilize a **Trie-based (prefix tree) state machine** combined with the **maximal munch principle**. As the buffer accumulates characters, the state machine evaluates the sequence. It continues consuming characters as long as they match a valid prefix path. Once a path breaks, the lexer yields the longest valid matched token.
* **Lexical Error Handling:** If the buffer yields no valid token match at all (an illegal character or malformed sequence), it registers a lexical error and skips forward to the next whitespace or known safe boundary to resume processing.
* **Comments:** The Lexer is responsible for identifying and safely discarding all comments `[ ... ]` before tokens are passed to the parser.

### 2.4 Parser

The Parser consumes the token stream generated by the Lexer and constructs the Abstract Syntax Tree (AST) strictly according to the Clime Extended Backus-Naur Form (EBNF) grammar.

* **AST Definition:** The **Abstract Syntax Tree (AST)** is a hierarchical tree of Python data objects (e.g., `dataclasses`) representing the underlying logical structure of the ruleset. Each node in the AST corresponds to a specific grammar construct (such as a `RuleNode`, `BinaryExpressionNode`, or `LiteralNode`) and holds references to its child components, completely stripping away superficial syntactic details like spaces, commas, or specific quotation marks.
* **Error Recovery:** The Parser must employ an error-recovery mechanism (such as token synchronization at rule boundaries) to continue parsing subsequent rules even if a syntax error is encountered in the current rule. All encountered errors must be accumulated and returned within the `ASTResult` object rather than raising immediate runtime exceptions.

### 2.5 Emitter

The Emitter traverses a validated AST object and translates it into the target JSON-IR format. It is responsible for injecting the correct `"kind"` and `"datatype"` structural properties into every node, unrolling syntactic sugar (like chained tag-list actions or numeric range comparisons) into their explicit underlying JSON-IR representations, and ensuring the final payload strictly conforms to the Climatomaton JSON-IR schema.

**Output Format:** The Emitter generates the JSON-IR representation natively as a **Python dictionary format**; it strictly does **not** output a serialized JSON string. It returns this dictionary encapsulated in an `IRResult` object.

## 3. CLI Tooling Specifications

The standalone CLI tool wraps the shared parsing library to provide local and automated validation capabilities.

### 3.1 Scope and Usage Constraint

**CRITICAL:** The CLI tooling is strictly isolated from the runtime operational architecture. It is **NOT** part of the Pluggable Rules Module (PRM) container, nor is it part of the Core Daemon process. Its use is exclusively reserved for local developer validation of `.rules` files and automated syntax checking within CI/CD workflows.

### 3.2 CLI Architecture & Execution

Because the underlying parsing library is strictly I/O agnostic, the CLI tool must implement the necessary file-loading wrappers to read `.rules` source files from disk and pass their contents (or iterators) into the library. To streamline local testing environments, the CLI tool must be configured as a direct entry point within the monorepo's `uv` workspace. This ensures the tool can be natively executed from the activated virtual environment path.

### 3.3 Interface and Exit Codes

To properly integrate with CI/CD pipelines and local terminal workflows, the CLI must adhere to standard execution behaviors:

* **Input Arguments:** Must accept a path to a specific `.rules` file, multiple paths to individual `.rules` files, or a directory containing multiple `.rules` files for batch validation.
* **Exit Codes:** Must return a `0` exit code on a fully successful validation pass, and a non-zero exit code (e.g., `1`) if any syntax errors are detected in the provided files.
* **Error Formatting:** Must parse the `errors` list returned by the library's `BaseResult` objects and format them into verbose, human-readable terminal output. This output should strictly map errors back to the exact line and column in the source `.rules` file to assist in rapid debugging.

### Comments & Discussion Points

* **State Machine Implementation:** The decision to use a Trie-based state machine for the Lexer is robust and solves the chunked-iterator problem elegantly. During the implementation phase, we will need to formalize the exact token enumeration types within the Python code (e.g., using `enum.Enum` or `enum.auto`) to map to the state machine's terminal nodes.
* **Result Polymorphism:** Leveraging Python's `dataclasses` for `BaseResult`, `ASTResult`, and `IRResult` will provide excellent type hinting out of the box, ensuring consumers of the library clearly understand the available fields at each pipeline stage.
